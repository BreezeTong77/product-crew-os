# PRD 内审 Demo

这个示例展示用户写完 PRD 后，Product Crew OS 如何先做产品自审，再决定是否召唤对应角色评审。

## 用户

```text
我写完 PRD 了，你先帮我做内审。
```

## 甜心教练-董董

```text
收到。我先不急着叫全员评审。

当前阶段判断：PM self-review。

我会先检查这份 PRD 的 9 个关键面：
1. 问题是否清楚。
2. 目标用户是否明确。
3. 业务目标是否可衡量。
4. 范围和非范围是否清楚。
5. 主流程是否闭环。
6. 异常和边界是否写到。
7. 数据和指标是否有口径。
8. 验收标准是否可测试。
9. 依赖和风险是否有 owner。

如果只是不清楚，我会先给你修订清单；如果涉及技术、数据、体验或测试阻塞，我再把对应角色叫进来。
```

## 如果发现技术风险

```text
这里我要把张工叫进来一次，因为这份 PRD 里有接口来源和状态回流问题。张工只看技术可行性和实现边界，不替业务决定优先级。
```

## 张工

```text
我先说结论，方向能做，但现在不能直接排期。研发会先问两个东西：任务清单是谁生成的、分配出去后状态怎么回流。你先把接口来源和任务状态补出来，我这边才好判断工作量。
```

## 主控收束

```text
张工这边先退场。我会把这条转成 review item：

- 问题：任务清单来源和状态回流不清楚。
- 影响：研发无法判断工作量和边界。
- Owner：PM 补充，Tech 复看。
- 处理方式：进入 PRD 修订清单。
```

## 输出 Artifact

```text
prd-review-notes.md
```

应该包括：

- 发现的问题。
- 阻塞项。
- 可后置项。
- 建议召唤角色。
- 下一版 PRD 修改清单。
