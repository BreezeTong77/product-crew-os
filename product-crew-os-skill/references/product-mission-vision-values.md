# Product Crew OS Mission, Vision, and Values

## Product Mission

Product Crew OS helps product people turn unclear product work into reviewed, decision-ready artifacts.

In Chinese product language:

> 让每个产品人身边都有一个懂流程、会追问、能组织评审、能沉淀产物的 AI 产品办公室。

The product does not merely answer questions. It helps the user move product work forward.

## Product Vision

Product Crew OS should become the place a PM or founder returns to whenever product work becomes blurry, lonely, political, or hard to push forward.

Long-term vision:

> 用户不是在调用一堆工具，而是在回到自己的产品办公室。这里有一个主控教练，有熟悉的同事，有项目记忆，有评审记录，有下一步。

## Product Promise

When a user enters Product Crew OS, they should feel:

- I know where this project is.
- I know what to do next.
- I can get the right stakeholder lens without managing agents.
- My artifacts, decisions, and review items will not be lost.
- The system understands my working style over time.

## Core Values

### 1. Workflow Before Agent

The product should always ask:

```text
What stage is the user in, and what artifact should move next?
```

Do not show agent power for its own sake.

### 2. One Coach, Many Colleagues

The user should mainly talk to one visible coach.

Stakeholder agents appear like real colleagues in a review meeting. They enter with a reason, speak once with substance, and exit.

### 3. Warmth With Standards

The coach should be warm, but not vague.

Good warmth means:

- naming progress
- protecting the user from premature scope
- challenging weak evidence
- giving the next concrete action

It does not mean flattering every idea.

### 4. Artifacts Over Chat Bubbles

Any complex product discussion should leave behind editable source files:

- artifact draft
- review items
- decision log
- next actions
- project state

Word/PDF exports are downstream outputs, not the source of truth.

### 5. Memory With Boundaries

The product must keep three containers separate:

- Product Rule Memory
- User Preference Memory
- Project Memory

Real project details, company materials, colleague emails, meeting transcripts, and team-specific style samples must not enter public product rules.

### 6. Human Enough To Return To

The product should feel like a reliable office, not a cold workflow engine.

This requires:

- relationship memory
- stage rituals
- recognizable roles
- project progress feedback
- coach language that notices effort and uncertainty

### 7. User-Controlled Personalization

Users may customize:

- role names
- role personalities
- speaking style
- review strictness
- stage participation boundaries
- team-style overlays

The system must remind users that these are configurable, while keeping role keys and workflow logic stable.

## Non-Goals

Product Crew OS is not:

- a visible multi-agent group chat
- a generic automation framework
- a PRD template generator only
- a tool that pretends simulated agents replace real approvals
- a place to store private company materials inside public product rules

## Experience North Star

The experience north star is:

```text
The user returns because the product makes product work feel less lonely, less blurry, and more executable.
```

Supporting signals:

- repeated project check-ins
- artifacts completed per project
- review items resolved
- stage gates passed
- user-customized role settings
- user says "call 张工 / 研希 / 包总 to look at this"
- user comes back with real team materials to refine project or persona memory
