# 真伪需求判别规则

## 1. 目的

Product Crew OS 必须帮助用户区分：

- 真需求。
- 弱需求。
- 伪需求。
- 待验证需求。

产品经理最容易犯的错，不是不会写需求，而是把“有人说想要”当成“值得做”。

## 2. 不算真需求的信号

以下信号只能作为线索，不能直接作为真需求：

- 用户说“我想要”。
- 客户说“有这个就好了”。
- 老板说“这个功能做一下”。
- 销售转述“客户很需要”。
- 问卷里选择“感兴趣”。
- 用户觉得“这个想法挺好”。
- 竞品有这个功能。

这些都要继续追问：

```text
什么时候发生？
上次发生是什么时候？
不解决会怎样？
现在怎么绕过去？
谁为这个问题付出了时间、钱或资源？
```

## 3. 真需求证据

强证据从高到低：

1. 观察到用户真实行为。
2. 用户已经付费或采购替代方案。
3. 用户已经搭建人工流程、脚本、表格或多工具 workaround。
4. 用户反复在真实项目中遇到同一问题。
5. 用户能讲出最近一次具体场景。
6. 多个目标用户独立提到同类问题。

弱证据：

- 用户表达兴趣。
- 用户说“可能会用”。
- 用户说“如果免费可以试试”。
- 用户说“以后应该有用”。

## 4. 四维评分

每个需求按 1-5 分打分。

### 4.1 发生频率

- 5：每天或一天多次。
- 4：每周。
- 3：每月。
- 2：每季度。
- 1：很少发生或只发生过一次。

### 4.2 痛感强度

- 5：阻塞关键工作，造成真实损失。
- 4：明显影响效率，需要大量绕路。
- 3：烦，但还能忍。
- 2：轻微不便。
- 1：可有可无。

### 4.3 替代方案

- 5：已经付费买替代产品或自建工具。
- 4：用多个工具拼流程。
- 3：有人工流程，但很麻烦。
- 2：偶尔搜索或临时处理。
- 1：没有试过解决。

### 4.4 付费/投入信号

- 5：已经在花钱或有预算。
- 4：明确说愿意付费并给出价格或采购方式。
- 3：口头说可能愿意付费。
- 2：只想免费用。
- 1：没想过付费或投入。

公式：

```text
需求真实性分 = 发生频率 x 痛感强度 x 替代方案 x 付费/投入信号
```

## 5. 判断标准

| 分数 | 判断 |
| --- | --- |
| 250+ | 强真需求，可以进入方案/PRD 前置 |
| 100-249 | 有希望，但需要补证据 |
| 50-99 | 弱需求，除非战略必要，否则不优先 |
| <50 | 伪需求或低价值需求，建议放弃或重定义 |

注意：

- 如果完全没有替代方案证据，即使用户说很痛，也不能直接判真需求。
- 如果没有付费或投入信号，商业优先级要谨慎。
- 如果只有一个用户提到，最多标记为线索，不标记为需求。

## 6. 主控教练输出格式

```text
需求判断：待验证 / 真需求 / 弱需求 / 伪需求
理由：
- 发生频率：
- 痛感强度：
- 替代方案：
- 付费/投入信号：
当前证据缺口：
下一步验证动作：
```

## 7. 阶段门

没有通过真伪需求判断时：

- 不进入 PRD。
- 不进入技术方案。
- 不进入排期。
- 不召唤研发做实现细节评审。

可以进入：

- 访谈。
- 证据盘点。
- 假设地图。
- fake door。
- concierge test。
- 手工试跑。
