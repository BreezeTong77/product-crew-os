# Project Intake Guard

`project_intake` 的任务是建项目，不是替用户把一个想法做成结论。

## 必须分开写

| 类型 | 能写什么 |
| --- | --- |
| 来自用户的事实 | 用户原话、已有文件、明确负责人、明确目标用户、明确目标 |
| 待验证假设 | 痛点、市场趋势、传播机制、竞品结论、需求真实性、用户偏好 |
| 当前未知 | 成功定义、数据基线、外部来源、是否愿意分享或付费 |

## 这一阶段能完成什么

- `project-card.md`
- 初始 `project-state.json`
- 持久化 route trace

## 这一阶段不能完成什么

- 给没有证据的需求打真实性分。
- 把市场热点、目标用户或痛点写成确认事实。
- 把 MVP、fake door、原型、结果库或 PRD 写成已决定路线。
- 因为文件写出来了，就报告 SOP、Skill、embedding 或子 Agent 已真实运行。

`爆款`、`增长`、`传播`、`分享`、`流量`、`出圈` 属于业务目标信号：Standard SOP 需要触发 Biz。没有真实委派能力时，必须留下 `pending` 或 `runtime_blocked` 记录。
